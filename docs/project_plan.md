
# Project Plan

## 1. 프로젝트 개요

### 1.1 배경

이 프로젝트는 종합 이커머스 도메인에서 발생하는 사용자 행동 로그를 기반으로 유저 여정을 분석하기 위한 end-to-end 분석 프로젝트이다.

사용자의 유입, 탐색, 상품 조회, 장바구니, 구매 전환, 이탈 흐름을 이벤트 로그 단위로 추적하고, 전환 병목과 행동 패턴을 파악하는 것을 목표로 한다. 기존 DB 설계, synthetic data 생성 및 MySQL 적재, 정합성 검증, SQL 분석 결과는 유지하되, 최종 포지션은 SQL, Python, Tableau를 연결한 분석 흐름으로 확장한다.

SQL은 원천 로그에서 퍼널, 전환, 리뷰 행동 관련 핵심 지표를 정확하게 산출하는 단계로 정의한다. Python은 단순 시각화를 넘어 Python 기반 세션 단위 분석 및 구매 전환 모델링, Python 기반 구매 전환 예측 및 실험 설계 단계로 확장한다. Tableau는 최종 대시보드로 전달 가능한 형태로 분석 결과와 인사이트를 정리하는 단계로 정의한다.

### 1.2 분석 범위

- 도메인: 종합 이커머스
- 분석 방식: 로그 기반 사용자 행동 분석, 구매 전환 예측, A/B 테스트 설계안 도출
- 주요 분석 대상: 사용자 이벤트 로그
- 주요 여정: 유입 -> 탐색 -> 상품 조회 -> 장바구니 -> 구매 -> 리뷰 작성
- 주요 관심사: 전환율, 이탈 지점, 사용자 행동 흐름, 이벤트 정합성, 구매 전환 예측, 리뷰 행동, Tableau 대시보드 설계

## 3. 도메인 정의

### 3.1 이커머스 유저 여정

종합 이커머스의 기본 유저 여정은 사용자가 서비스에 유입된 뒤 상품을 탐색하고, 상품 상세를 확인하며, 장바구니 또는 구매 단계로 이동하는 흐름으로 정의한다.

초기 기준 여정은 다음과 같다.

1. 서비스 유입
2. 상품 목록 또는 검색 결과 탐색
3. 상품 상세 조회
4. 장바구니 담기
5. 주문 시작
6. 결제 완료

### 3.2 주요 엔티티

| 엔티티 | 설명 | 현재 상태 |
|---|---|---|
| `users` | 서비스를 이용하는 사용자 | 역할 정의 완료, 속성 초안 완료 |
| `sessions` | 사용자의 방문 단위 | 역할 정의 완료, 속성 초안 완료 |
| `categories` | 상품 분류 체계 | 역할 정의 완료, 속성 초안 완료 |
| `products` | 판매 상품 | 역할 정의 완료, 속성 초안 완료, 이벤트 `item_id`와 `product_id` 매핑 정리 |
| `orders` | 주문 또는 결제 결과 | 역할 정의 완료, 속성 초안 완료 |
| `order_items` | 주문에 포함된 개별 상품 | 역할 정의 완료, 속성 초안 완료 |
| `reviews` | 구매 이후 작성된 상품 리뷰 | 역할 정의 완료, 속성 초안 완료 |
| `event_logs` | 사용자의 행동 로그 단위 | 이벤트 정의 완료, 공통 필드 정의 완료 |

### 3.3 주요 이벤트

초기 이벤트 설계는 GA4 스타일 이벤트 설계를 따른다. 이벤트는 `event_name`, `event_timestamp`, `user_id`, `session_id`, 이벤트 파라미터를 중심으로 정의한다.

| 이벤트 | 설명 | 비고 |
|---|---|---|
| `session_start` | 세션 시작 | 유입 분석 기준 |
| `search` | 검색 실행 | 검색 행동 분석 |
| `view_item_list` | 상품 목록 조회 | 탐색 단계 |
| `view_item` | 상품 상세 조회 | 상품 관심도 기준 |
| `add_to_cart` | 장바구니 담기 | 구매 의도 기준 |
| `begin_checkout` | 주문 시작 | 구매 퍼널 진입 |
| `purchase` | 구매 완료 | 최종 전환 |
| `review_write` | 리뷰 작성 | 구매 이후 행동 분석 |

### 3.4 주요 지표

- 세션 수
- 사용자 수
- 상품 조회 수
- 장바구니 전환율
- 체크아웃 전환율

## 5. 현재 설계

### 5.1 이벤트 설계

GA4 스타일 이벤트 설계를 채택한다.

기본 방향은 다음과 같다.

- 사용자 행동을 이벤트 단위로 기록한다.
- 이벤트 이름은 행동을 설명하는 명확한 동사형 또는 GA4 권장 이벤트명을 사용한다.
- 이벤트 공통 필드는 `event_name`, `event_timestamp`, `user_id`, `session_id`를 중심으로 구성한다.
- 상품, 주문, 검색어, 유입 경로 등은 이벤트 파라미터로 관리한다.
- 퍼널 분석이 가능하도록 주요 단계 이벤트를 명시적으로 정의한다.

### 5.2 데이터 모델

초기 데이터 모델은 이벤트 로그 중심으로 시작하되, 이벤트 분석에 필요한 1차 참조 엔티티를 함께 도출한다.

| 테이블 | 목적 | 현재 상태 |
|---|---|---|
| `event_logs` | 사용자 행동 이벤트 원천 로그 | 이벤트 정의 완료, 공통 필드 정의 완료, 이벤트별 파라미터 정의 완료 |
| `users` | 사용자 속성 정보 | 엔티티 도출 완료, 역할 정의 완료, 속성 초안 완료 |
| `sessions` | 사용자 방문 단위 정보 | 엔티티 도출 완료, 역할 정의 완료, 속성 초안 완료 |
| `categories` | 상품 카테고리 정보 | 엔티티 도출 완료, 역할 정의 완료, 속성 초안 완료 |
| `products` | 상품 속성 정보 | 엔티티 도출 완료, 역할 정의 완료, 속성 초안 완료 |

현재 단계에서는 물리적 테이블 설계가 아니라 논리적 엔티티와 주요 속성을 정리한다. 엔티티 간 관계 분석은 1차 완료했으며, 정합성 검증 항목은 2차 초안까지 작성했다. 논리적 테이블 설계 1차를 바탕으로 `docs/table_specification.md`에 테이블별 역할, 컬럼, 키, nullable 여부, MySQL 데이터 타입 후보를 정리했다.

MySQL 기준 물리 스키마는 `sql/schema.sql`에 작성했다. 소규모 `sql/seed_data.sql`은 쿼리 구조 검증용으로 유지하고, 현재 주요 SQL 분석 결과는 `sql/seed_synthetic_data.sql` 기반 synthetic data로 갱신한 상태다.

### 5.3 정합성 검증 계획

정합성 검증은 분석 이전 단계에서 수행한다.

1차 검증 규칙은 `data_quality_rules.md`에 문서화한다.

초기 검증 항목은 다음과 같다.

- 필수 필드 누락 여부
- 이벤트명 허용 목록 위반 여부
- 사용자 ID 또는 세션 ID 누락 여부
- `event_logs.user_id`와 `sessions.user_id` 일치 여부
- 이벤트 타임스탬프 역전 여부
- 동일 이벤트 중복 발생 여부
- 구매 이벤트의 주문 ID 누락 여부
- 퍼널 순서 위반 여부

- SQL 분석 결과 1차 문서화 완료
- synthetic data 생성 및 MySQL 적재 완료
- synthetic data 기준 row count 확인 완료
- synthetic data 기준 18개 정합성 검증 모두 위반 0건 확인 완료
- 핵심 질문 3개에 대한 SQL 분석을 synthetic data 기준으로 재실행 완료
- `docs/analysis_results.md`를 synthetic data 기준으로 업데이트 완료

- `with_checkout` 모델은 `begin_checkout_count`, `has_begin_checkout`을 포함했으며 F1 0.9032, ROC-AUC 0.9838을 보였다.
- `without_checkout` 모델은 구매 직전 행동인 `begin_checkout_count`, `has_begin_checkout`을 제외했으며 F1 0.8889, ROC-AUC 0.9834를 보였다.
- `begin_checkout`을 제외해도 성능이 크게 유지되어, 결제 이전 세션 행동 패턴만으로도 구매 전환 여부를 일정 수준 이상 구분할 수 있음을 확인했다.
- `without_checkout` 모델 기준 `event_count`, `has_add_to_cart`가 주요 양의 신호로 나타났다.
- 일부 세부 행동 횟수 변수는 `event_count`, `has_add_to_cart` 등과 정보가 중복되면서 계수 부호가 분산된 결과로 해석했다.
- 초기 모델링 과정에서 `session_duration_minutes`가 `purchase`, `review_write` 이벤트를 포함할 수 있는 target leakage 가능성을 확인했다.
- 이후 `sql/session_level_features.sql`에서 `session_duration_minutes`를 `purchase`, `review_write` 제외 기준으로 재정의했다.
- 수정 후에도 모델 성능이 유지되어, 보수적으로 feature 정의를 조정한 뒤에도 모델링 흐름이 안정적으로 작동함을 확인했다.
- SQL 퍼널 분석과 Logistic Regression baseline 결과를 바탕으로 `docs/ab_test_design.md`에 A/B 테스트 설계안을 1차 정리했다.
- SQL 분석, Python 모델링, A/B 테스트 설계 결과를 하나의 흐름으로 전달하기 위한 `docs/tableau_dashboard_design.md`를 1차 정리했다.
- `README.md`를 GitHub 포트폴리오 제출용으로 정리 완료

### 6.8 현재 한계

- 현재 주요 분석 결과는 `sql/seed_synthetic_data.sql` 기반 synthetic data다.
- synthetic data는 SQL 쿼리 구조와 분석 흐름 검증에는 적합하지만, 실제 서비스 인사이트로 일반화할 수는 없다.
- Logistic Regression baseline 성능 역시 실제 서비스 예측 성능으로 일반화하지 않는다.
- 모델링 목적은 실제 운영 모델 구축이 아니라, 세션 단위 feature 설계와 구매 전환 예측 분석 파이프라인을 검증하는 것이다.
- 실제 인사이트 도출을 위해서는 실제 로그 데이터 또는 더 현실적인 생성 규칙을 반영한 데이터가 필요하다.
- 현재 단계의 A/B 테스트는 실제 실험 수행 결과가 아니라, synthetic data 기반 분석 흐름에서 확인한 퍼널 및 전환 지표를 바탕으로 도출하는 설계안이다.
