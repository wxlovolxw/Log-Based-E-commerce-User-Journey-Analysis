이벤트 정의.

이벤트란? 

    ** 사용자가 특정 시점에 수행한 행동

    서비스 관점에서는 서비스에서 발생하는 행동을 중심으로 이벤트를 정의.

    ex) 상세 페이지 보기, 찜하기, 장바구니 담기, 쿠폰 받기, 구매하기

    분석 관점에서는(GA4, Amplitude와 같은 분석 도구) 서비스가 달라도 재사용이 가능한 형태로 표준화. 

        * 이벤트 로그 분석 툴 -> GA4, Amplitude, Mixpanel. 대부분의 기업들이 툴을 사용.

            사용자 행동 수집 -> 저장 -> 분석까지.

            직접 만드려면 수집 서버, 적재, 세션관리, 분석화면, 대시보드 까지 비용이 비쌈

    ex) search, page_view, add_to_chart

    -> 서비스 관점은 특정 쇼핑몰에 특화, 분석 관점은 실무 분석도구 스타일


------------------------------------------------------------------------------------------------

주요 이벤트

    어떤 비즈니스 질문에 답하고 싶은가?

    1) 사용자는 어디서 가장 많이 이탈하는가

        방문 -> 상품조회 -> 장바구니 -> 구매

    2) 검색하는 사람들의 구매율이 더 높은가?

        search

    3) 상품은 많이 보는데 왜 안사는가?

        view_item

    4) 장바구니에 담았지만 구매하지 않는 이유는 무엇인가?

        add_to_chart, begin_checkout, purchase

    -> *** 핵심 질문 설정

    - 사용자들이 구매과정의 어느 단계에서 이탈하는가?

    - 어떤 행동이 구매 전환과 가장 밀접한 관련이 있는가?

    - 구매 이후 사용자 참여는 어떠한가?


    * 분석 질문 -> 필요한 행동 -> 이벤트

    * 유저들의 행동 여정

        1. 유입(session_start) 
    
        2. 탐색(search, view_item_list) 
    
        3. 관심(view_item) 
    
        4. 구매의도(add_to_chart)  
    
        5. 결제(begin_checkout) 
    
        6. 구매(purchase)

    -> 최소한의 이벤트로 전 과정을 보기 위해 해당 6개의 이벤트를 사용

    (각 유저 여정 단계를 대표하는 최소의 이벤트 선택)

    session_start
    search
    view_item_list
    view_item
    add_to_cart
    begin_checkout
    purchase
    review_write

- 이벤트별 상세 정의란?

이벤트명, 발생 조건, 분석 조건, 필수 파라미터

이 행동 로그를 언제, 어떤 이름으로, 어떤 정보와 함께 남길것인가

ex) view_item : 사용자가 상세 페이지를 봤을때 view_item 이벤트를 남긴다.

    어떤 상품을 봤는지 확인해야 하기 때문에 item_id가 필요하다.

    나중에 카테고리별 전환율도 확인하고 싶다면 category_id도 있으면 좋을 것 같다.

이벤트에 필요한 항목

    event_name, 이벤트 설명, 발생 조건, 분석 목적, 필수 파라미터, 선택 파라미터, 정합성 검증 포인트

    파라미터를 통해 행동의 세부 정보를 얻음

ex) search 이벤트

    event_name : search

    발생 조건 : 사용자가 검색어를 입력하고 검색을 실행했을 때

    필수 파라미터 : search_term

    선택 파라미터 : search_result_count, sort_option, filter_options

    -> 검색어가 있어야 무엇을 검색했는지 확인 가능하기 때문에 search_term이 필수

    * search_term : 사용자의 검색어

ex) view_item 이벤트

    event_name : view_item
    
    발생 조건 : 사용자가 상품 상세 페이지를 봤을 때

    필수 파라미터 : item_id가

    선택 파라미터 : category_id, price, item_brand

이벤트 상세 정의

< 유입 -> 탐색 -> 관심 >

1. session_start

    사용자가 쇼핑몰에 방문함. (방문자수 확인)

    -> 여기서 추가로 확인하고 싶은 부분?

        어떻게 들어왔는지. 광고페이지, 구글검색, 직접접속

        파라미터 후보 : user_id, session_id, traffic_source, device_type


2. search

    사용자가 무언가를 찾으려고 함. (구매 의도가 높은 행동 확인)

    -> 뭘 검색했는가?

        search_term(필수)


3. view_item_list

    상품 목록을 봄. 

        사용자는 검색 -> 상품목록 -> 상품상세 로 넘어감

    -> 어떤 목록을 봤는가?

        item_list_id, item_list_name


4. view_item

    상품 상세를 봄 (사용자의 관심이 발생한 순간)

    -> 어떤 상품을 봤는가?

        item_id, item_name

< 구매 >

5. add_to_cart

    사용자가 상품을 장바구니에 담음. (단순 관심이 아닌 구매 의도)

    -> 상품 상세를 본 사람중 실제 구매 의도를 보인 비율

        item_id, quantity


6. begin_checkout

    사용자가 주문/결제 절차를 시작함.

    add_to_cart, purchase만 있으면 사용자가 어디에서 포기했는지를 확인하기 쉽지 않음.

    결제 페이지 까지 가고 최종 결제를 했는지 안했는지를 구분하기 위함.

    add_to_cart -> begin_checkout -> purchase

        checkout_id, items, value, currency


7. purchase

    구매/ 결제가 완료되었다. (최종 전환 이벤트)

    단순히 샀다 가 아닌 검증해야 하는 항목들이 추가적으로 필요

        order_id, items, value, currency


< 구매 이후 행동 이벤트 >

8. review_write

    사용자가 구매한 상품에 대해 리뷰를 작성함

    구매한 사용자 중 얼마나 적극적으로 참여하는가?

        order_id, item_id, review_id, rating

        선택 정보 : review_length, has_photo, has_video

        추가적으로 review_test를 통한 감성분석 가능

공통 이벤트 필드 - 모든 이벤트가 반드시 가져야 하는 정보

    event_id        이벤트 고유 ID      중복 검증, 개별 이벤트 식별

    event_name      이벤트 이름         어떤 행동인지 구분

    event_timestamp 이벤트 발생 시각     순서 분석, 세션 흐름 분석

    user_id         사용자 ID           사용자별 행동 분석

    session_id      세션 ID             방문 단위 퍼널 분석

    event_date      이벤트 발생 날짜      일자별 집계 편의용


이벤트 별 필수 파라미터 정의

    -> 각 이벤트가 자기 의미를 가지기 위해 반드시 필요한 값은 무엇인가?

    session_start   ->  X

    search          ->  search_term

    view_item_list  ->  item_list_id(필수), item_list_name(선택)

    view_item       ->  item_id(필수), item_name(선택)

    add_to_cart     ->  item_id, quantity(필수), item_name(선택)

    begin_checkout  -> checkout_id, value(필수), items(선택)

    purchase        -> order_id, value(필수), items(선택)

    review_write    -> review_id, order_id, item_id, rating

    *** 이벤트 로그에는 행동 발생 사실과 연결 식별자를 중심으로 기록한다.

    이름/상세설명/주문상품상세는 별도 엔티티 또는 참조 테이블에서 관리한다.

* 엔티티 정의

    이벤트와 엔티티의 차이점

        이벤트 -> 일어난 일. 사용자가 서비스 안에서 한 행동의 기록한다

            ex) 사용자가 검색했다.

            상품 목록을 봤다.

            장바구니에 담았다.

            -> 동작 중심. '무슨 일이 일어났는가?' 에 대한 답

        엔티티 -> 그 일이 일어난 대상. 이벤트가 참조하는 대상, 객체, 개념

            ex) view_item : 사용자가 상품을 조회했다.

            이때의 엔티티 -> 사용자, 상품, 세션(세션안에서 발생했기 때문)

        * view_item이라는 이벤트는 상품 조회라는 행동. 

        * 이 행동이 참조하는 엔티티는 user, session, product

        이벤트      발생한 행동         검색함, 상품 봄, 구매함 (동사)

        엔티티      행동과 관련된 대상   사용자, 세션, 상품, 주문 (명사)

-----------------------------------------------------------------------------------------------

엔티티 도출의 기준

    이벤트들이 참조하는 대상이 무엇인가?

    해당 대상이 별도로 관리될 만큼 중요한가?

    -> 여러 이벤트에 반복적으로 등장 하였는지, 속성 정보를 따로 관리해야 하는지, 분석 질문과 연결되는지

    이벤트와 이벤트를 자연어로 정의하면,

        session_start   사용자가 세션을 시작한다.

        search          사용자가 검색한다.

        view_item_list  사용자가 상품 목록을 본다.

        view_item       사용자가 상품 상세를 본다.

        add_to_cart     사용자가 장바구니에 담는다.

        begin_checkout  사용자가 결제를 시작한다.

        purchase        사용자가 구매한다.

        review_write    사용자가 리뷰를 작성한다.

    ** 1차로 필요한 엔티티

    1) users - 사용자 엔티티

        누가 한 행동인지. -> user_id를 기준으로 이벤트 로그와 연결

            사용자는 어디서 이탈하는가?

            구매한 사용자와 구매하지 않은 사용자는 무엇이 다른가?

            구매 이후 사용자는 어떤 행동을 하는가?


    2) session - 세션 엔티티

        한 번의 방문 단위. -> 같은 사용자가 하루에 여러번 방문할 수 있음.

        같은 user_id라도 session_id가 다를수 있음.

        ex) S001: search -> view_item -> add_to_cart -> purchase

            S002: search -> view_item에서 이탕

        -> 세션별로 여정을 확인해야 함.


    3) product - 상품 엔티티

        view_item_list, view_item, add_to_cart, purchase, review_write 에서 계속 등장

        각 이벤트마다 넣는거 보다 products에서 관리하는게 좋음.

        -> product_id, product_name, category_id, price, brand 와 같은 정보

            어떤 카테고리에서 조회는 많은데 구매가 적은가?

            어떤 가격대 상품이 장바구니 이탈이 많은가?

            어떤 상품군이 리뷰 작성률이 높은가?


    4) categories - 카테고리 엔티티 (상품의 분류)

        카테고리 별 퍼널 분석을 자주 하게 될 경우 필요.

        products 안에 문자열 컬럼으로 둬도 됨. (상의/하의/신발/가방/전자기기/생활용품 등)

            카테고리별 조회 -> 장바구니 -> 구매 전환율

            카테고리별 이탈 구간

            카테고리별 리뷰 작성들

        -> categories는 권장, products는 필수.

    
    5) orders - 주문 엔티티

        purchase 이벤트 발생시 주문이 생김.

        이벤트는 "구매했다" 라는 행동, orders는 그 결과로 생긴 주문 기록

        ex) purchase 이벤트 발생 -> order_id = O001 생성

        주문에는 다음과 같은 정보가 붙을 수 있음.

            order_id, user_id, order_timestamp, total_amount, payment_method, order_status

            구매 금액은 얼마인가?

            주문 상태는 완료인가 취소인가?

            사용자별 구매 횟수는 몇 번인가?

            세션당 구매 전환율은 얼마인가?

    
    6) order_items - 주문 상품 엔티티

        보통 주문 하나에는 여러 상품이 들어갈 수 있음

        ex) order_id = O001
            
            - product_id = P001

            - product_id = P002

            - product_id = P003

            -> 주문과 상품은 1:1 관계가 아님. 그렇기 때문에 중간에 order_items가 필요함.

                order_id, product_id, quantity, item_price, discount_amount

        orders -> 주문 단위

        order_items -> 주문 안의 개별 상품 단위


    7) reviews - 리뷰 엔티티

        review_write 이벤트 발생시 리뷰가 생김

        -> review_write = 리뷰를 작성했다는 행동

        reviews = 작성된 리뷰 자체

            review_id, user_id, product_id, order_id, rating, review_text, created_at

        구매 이후 사용자는 어떤 행동을 하는가?


    ** 1차 엔티티 정리

    users       사용자별 여정, 전환, 재방문 분석

    sessions    방문 단위 퍼널과 이탈 분석

    products    상품 조회, 장바구니, 구매, 리뷰 분석

    categories  카테고리별 전환율/이탈 분석

    orders      구매 결과와 주문 단위 분석

    order_items 주문 안의 개별 상품 분석

    reviews     구매 이후 행동 분석

    -> 선택 후보

        search_terms, carts, cart_items, traffic_source, coupons, promotions 등

각 이벤트의 엔티티 참조 확인 + 설명을 위한 파라미터

1. session_start

    사용자가 서비스를 방문해서 세션을 시작했다.

    사용자, 세션 -> users, sessions

    파라미터 - traffic_source, medium, campaign, device_type, platform, landing_page


2. search
    
    사용자가 검색했다.

    사용자, 세션-> users, session

    * 검색어는 엔티티가 아닌 이벤트 파라미터로 설정.

    -> 이후의 이벤트를 묶는 단위가 아닌 설명값에 가깝기 때문에 파라미터로 설정.

    * 이벤트가 늘 새로운 엔티티를 만드는 것은 아니다.

    파라미터 - search_term, page_number


3. view_item_list

    사용자가 상품 목록 화면을 봤다. -> 특정 상품 하나가 아닌 여러 상품이 나열된 목록 화면

    사용자, 세션, 카테고리 -> users, session, categorie

    파라미터 - list_id, page_number


4. view_item

    사용자가 특정 상품 상세 페이지를 봤다.

    사용자, 세션, 상품 -> users, sessions, products

    (누가, 어떤방분에서, 어떤 상품을 봤는가)

    파라미터 - 없음 / position, list_id


5. add_to_cart

    사용자가 특정 상품을 장바구니에 담았다. (구매 의도가 강한 행동)

    사용자, 세션, 상품 -> users, sessions, product

    * 장바구니 상태를 더 상세히 보려면 carts 테이블을 별도로 생성

    -> 현재로서는 장바구니에 담았다 라는 행동 자체가 중요함.

    파라미터 - quantity, price


6. begin_checkout

    사용자가 결제 절차를 시작했다. (퍼널분석에서 매우 중요)

    사용자, 세션 -> users, sessions

    파라미터 - items_count, total_value


7. purchase

    사용자가 구매를 완료했다. (주문이 생성되고 구매가 완료 됨.)

    사용자, 세션, 구매 -> users, sessions, orders

    * purchase 이벤트에서는 order_id를 남기고, 상세 정보는 orders 테이블에서 관리한다.

    파라미터 - total_value, payment_method


8. review_write

    사용자가 구매 이후 리뷰를 작성함. (구매 전환 이후의 행동을 보는 이벤트.)

    구매 이후 사용자가 어떤 행동을 하는가?

    사용자, 세션, 제품, 구매, 리뷰 -> users, sessions, product, orders, review

    파라미터 - rating, review_length


------------------------------------------------------------------------------------------------

엔티티별 역할

    users       사용자를 식별하고 사용자 단위의 행동 여정을 연결하는 기준 엔티티

    sessions    사용자의 한 번의 방문 단위를 나타내며, 세션 단위 퍼널과 이탈 분석의 기준 엔티티

    categories  상품 분류 정보를 관리하며, 카테고리별 조회/전환/이탈 분석에 사용되는 참조 엔티티

    products    사용자가 조회/장바구니/구매/리뷰 작성하는 상품 정보를 관리하는 핵심 엔티티

    orders      구매 완료로 생성된 주문 단위 정보를 관리하는 엔티티

    order_items 하나의 주문에 포함된 개별 상품 정보를 표현하는 연결 엔티티

        -> orders와 products 사이의 관계를 풀어주는 엔티티

    reviews     구매 이후 사용자가 작성한 리뷰 정보를 관리하는 엔티티

------------------------------------------------------------------------------------------------

엔티티 별 주요 속성

    엔티티가 어떤 정보를 가져야 하는지.

    1. users : 사용자 단위 여정을 연결하는 기준 엔티티

        user_id         사용자 식별 * 핵심

        signup_date     가입일

        user_type       신규/기존/회원/비회원 등 사용자 유형

        gender          성별, 선택 가능

        age_group       연령대, 선택가능

        -> 사용자의 여러 세션과 행동의 연결이 필요

        users.user_id -> event_log.user_id -> sessions.user_id, orders.user_id


    2. sessions : 한번의 방문 단위 (퍼널 분석의 기준)

        session_id          세션 식별자

        user_id             세션을 발생시킨 사용자

        session_start_time  세션 시작 시간

        session_end_time    세션 종료 시각

        traffic_source      유입 출처

        medium              유입 매체

        campaign            캠페인명

        device_type         기기 유형

        platform            웹/앱 구분

        landing_page        첫 진입 페이지

        -> 한 세션 안에서,

        session_start -> search -> view_item -> add_to_cart -> purchase

        까지 갔는지 확인하고 싶음.


    3. categories : 상품 분류 기준

        category_id         카테고리 식별자

        category_name       카테고리명

        (parent_category_id  상위 카테고리, 선택가능) -> 대분류 / 중분류 구조


    4. products : 상품 정보를 관리

        product_id          상품 식별자

        product_name        상품명

        category_id         카테고리 식별자

        brand               브랜드

        price               현재 상품 가격

        created_at          상품 등록일

        is_active           판매 활성 상태

        -> 상품은 여러 이벤트에서 계속 참조되기 때문에 핵심 엔티티임


    5. orders : 구매 완료 후 생성된 주문 단위

        order_id            주문 식별자

        user_id             주문한 사용자

        session_id          주문이 발생한 세션

        order_timestamp     주문 시각

        total_value         주문 총액

        payment_method      결제 수단

        order_status        주문 상태

        -> 취소/환불/완료 여부를 위해 order_status가 필요함


    6. order_items : 주문 안의 개별 상품 단위

        order_item_id       주문 상품 행 식별자

        order_id            주문 식별자

        product_id          상품 식별자

        quantity            구매 수량

        item_price          구매 당시 상품 단가

        discount_amount     할인 금액, 선택 가능

        -> 상품 가격이 변할 수 있으니 product.price와 order_items.item_price는 따로 남겨두는게 좋음.


    7. reviews : 구매 이후 작성된 리뷰

        review_id           리뷰 식별자

        user_id             리뷰 작성자

        product_id          리뷰 대상 상품

        order_id            리뷰와 연결된 주문

        rating              평점

        review_length       리뷰 길이

        created_at          리뷰 작성 시각

        -> 현재는 본문 텍스트 까지 분석하는게 목적은 아님


------------------------------------------------------------------------------------------------

엔티티 관계 확인 (직접 관계 -> PK-FK 연결)

    users 1:N sessions	        한 사용자는 여러 세션을 가질 수 있다

        sessions.user_id -> users.user_id

        한 사용자는 여러 세션을 가질 수 있다. 하나의 세션은 한 사용자에게 속한다.

    sessions 1:N event_logs	    한 세션 안에는 여러 이벤트가 발생할 수 있다

        event_logs.session_id -> sessions.session_id
        
        한 세션 안에는 여러 이벤트가 발생할 수 있다. 하나의 이벤트는 하나의 세션에 속한다.

    users 1:N event_logs	    한 사용자는 여러 이벤트를 발생시킬 수 있다

        event_logs.user_id -> users.user_id

        한 사용자는 여러 이벤트를 발생시킬 수 있다. 하나의 이벤트는 한 사용자에게 속한다.

        * event에 user_id가 없어도, sessions를 통해 user를 확인할 수 있지만, 편의상 사용하도록 한다.

    categories 1:N products	    한 카테고리는 여러 상품을 가진다

        products.category_id -> categories.category_id

        상품은 어떤 카테고리에 속해있고, 카테고리는 여러 상품을 가질수 있다.

        (카테고리별 행동 분석)

        -> event_logs에 category_id를 저장할수 있지만 products.category_id와 중복이 발생

        -> 따라서 category_id는 products를 통해 확인하는 방식 선택

    products 1:N event_logs	    한 상품은 여러 조회/장바구니 이벤트에서 참조될 수 있다

        상품 관련 이벤트에는 product_id가 포함된다.

        ex) view_item을 보면, U001 사용자가 S001 세션에서 P001 상품을 봤다.

        event_logs.product_id -> products.product_id

        하나의 상품은 여러 이벤트에서 반복적으로 등장할 수 있음.

        ** 단, 모든 event_logs가 product_id를 갖는건 아님. 상품을 참조하지 않는 이벤트도 존재. 

    users 1:N orders	        한 사용자는 여러 주문을 할 수 있다

        orders.user_id -> users.user_id

        한 사용자는 여러 주문을 할 수 있다. 하나의 주문은 한 사용자에게 속한다.

    sessions 1:N orders	        한 세션에서 여러 주문이 발생할 수 있다

        orders.session_id -> sessions.session_id

        한 세션에서 주문이 발생할 수 있다. 하나의 주문은 특정 세션에서 발생한다.

        ** 한 세션에서 주문이 0개일 수도, 1개일 수도, 여러개 일 수도 있다.

    orders 1:N order_items	    한 주문은 여러 상품 행을 가진다

        order_items.order_id -> orders.order_id

        한 주문은 여러 주문 상품 행을 가질 수 있다. 하나의 주문 상품 행은 하나의 주문에 속한다.

        ** users -> sessions -> orders -> order_items

        (사용자별 구매 여부, 세션별 구매 전환율, 주문당 상품 수, 상품/카테고리별 매출)

    products 1:N order_items	한 상품은 여러 주문 상품 행에 포함될 수 있다

        order_items.product_id -> products.product_id

        하나의 상품은 여러 주문 상품 행에 포함될 수 있다. 하나의 주문 상품 행은 하나의 상품을 가리킨다.

    users 1:N reviews	        한 사용자는 여러 리뷰를 작성할 수 있다

        reviews.user_id -> users.user_id

        한 사용자는 여러 리뷰를 작성할 수 있다. 하나의 리뷰는 한 사용자가 작성한다.

    products 1:N reviews	    한 상품은 여러 리뷰를 받을 수 있다

        reviews.product_id -> products.product_id

        한 상품은 여러 리뷰를 받을 수 있다. 하나의 리뷰는 하나의 상품에 대해 작성된다.

    orders 1:N reviews	        한 주문에서 여러 리뷰가 작성될 수 있다

        reviews.order_id -> orders.order_id

        한 주문에서 여러 리뷰가 작성될 수 있다. 하나의 리뷰는 특정 주문을 바탕으로 작성된다.

정합성 검정 1차 - 테이블 간 연결된 ID들이 서로 모순이 없이 맞는지 확인하는 규칙을 정하는 단계

- 참조 무결성
- 중복 저장된 정보의 일치 여부
- 비즈니스 규칙 검증

-> users, sessions, event_log, orders, purchase

    1. 사용자 - 세션 - 이벤트 정합성 (세션은 실제 사용자에게 속해야 한다.)

        users, sessions, event_logs

        ** event_logs.session_id는 sessions.session_id에 존재해야 한다.

        (이벤트 로그에 기록된 세션은 실제 session 테이블에 존재하는 세션이여야 한다.)


    2. event_logs.user_id 와 sessions.user_id의 일치 (이벤트는 실제 세션에 속해야 한다.)

        event_logs.user_id = sessions.user_id


    3. sessions.user_id는 users.user_id에 존재해야 한다. (이벤트의 사용자와 세션의 사용자가 일치해야 한다.)

        sessions.user_id는 users.user_id에 존재해야 한다.


    4. 주문 - 세션 - 사용자 정합성

        orders.user_id는 users.user_id에 존재해야 한다. (주문 사용자는 실제 사용자여야 한다.)

        orders.session_id는 sessions.session_id에 존재해야 한다. (주문은 실제 세션에서 발생해야 한다.)

        orders.user_id = sessions.user_id 이어야 한다. (주문 사용자와 세션 사용자가 일치해야 한다.)


    5. 주문 이벤트와 주문 테이블 정합성

        event_logs에서 event_name = 'purchase'인 경우,

        -> event_logs.order_id는 orders.order_id에 존재해야 한다. (구매 이벤트는 실제 주문과 연결되어야 한다.)

        purchase 이벤트의 user_id/session_id는 orders의 user_id/session_id와 일치해야 한다.

        (구매 이벤트 사용자와 주문 사용자가 일치해야 한다.)


    *** event_logs, sessions, orders에 user_id가 반복해서 들어간다.

    그래서 같은 session_id/order_id를 기준으로 user_id가 서로 일치해야 한다.

    -> 단순히 ID가 존재하는지만 확인하는 것이 아니라, 
    
    '이 이벤트/주문이 정말 그 사용자의 세션에서 발생했는가?' 까지 확인해야 함.

정합성 검증 2차 - categories, products, order_items, reviews

    1. 상품 - 카테고리 정합성

        products - categories

        products.category_id 는 categories.category_id에 존재해야 한다.

        -> 상품은 실제 존재하는 카테고리에 속해야 한다.

    
    2. 이벤트 - 상품 정합성

        event_logs - products

        상품 관련 이벤트의 product_id는 products.product_id에 존재해야 한다.

        * 모든 이벤트가 product_id를 갖는 것은 아님.

        -> 상품 관련 이벤트는 실제 상품을 참조해야 한다.

    
    3. 주문상품 - order_items 정합성

        order_items.order_id 는 orders.order_id에 존재해야 한다.

        -> 주문 상품 행은 실제 주문에 속해야 한다.


    4. 주문상품 - 상품 정합성

        order_items.product_id는 products.product_id에 존재해야 한다.

        -> 주문 상품 행은 실제 상품을 참조해야 한다.


    5. 주문 총액 정합성

        orders.total_value는 order_items 기준 계산 금액과 일관되어야 한다.

        단, 배송비/쿠폰/포인트/세금 정책이 추가되면 계산식은 조정될 수 있다.

        -> 주문 총액은 주문 상품 행 기준 금액과 일관되어야 한다.


    6. 리뷰 - 사용자 정합성

        reviews.user_id 는 users.user_id에 존재해야 한다.

        -> 리뷰 작성자는 실제 사용자여야 한다.
    

    7. 리뷰 - 상품 정합성

        reviews.product_id 는 products.product_id에 존재해야 한다.

        -> 리뷰 대상 상품은 실제 상품이어야 한다.


    8. 리뷰 - 주문 정합성

        reviews.order_id 는 orders.order_id에 존재해야 한다.

        -> 리뷰는 실제 주문에 연결되어야 한다.


    9. 리뷰 작성자와 주문자 일치

        -> 리뷰 작성자는 해당 주문의 구매자여야 한다.


    10. 리뷰 상품이 실제 구매 상품인지 검증

        단순히 reviews.order_id가 orders에 있고, reviews.product_id가 products에 있다고 충분하지 않음.

        이 조합이 실제 주문 상품에 있어야 함.

        (reviews.order_id, reviews.product_id) 

        exists in

        (order_items.order_id, order_items.product_id)

        -> 리뷰는 실제 구매한 상품에 대해서만 작성되어야 한다.


    11. 리뷰 이벤트와 리뷰 테이블 정합성

        event_logs에서 event_name = 'review_write'인 경우,

        event_logs.review_id는 reviews.review_id에 존재해야 한다.

        사용자/주문/상품 도 일치해야 한다.

        -> 리뷰 작성 이벤트는 실제 리뷰와 연결되어야 한다.

        -> 리뷰 이벤트 사용자와 리뷰 작성자가 일치해야 한다.

        -> 리뷰 이벤트 상품과 리뷰 대상 상품이 일치해야 한다.

        -> 리뷰 이벤트 주문과 리뷰 연결 주문이 일치해야 한다.


    * orders.total_value는 order_items 기준 계산 금액과 일관되어야 한다.

    단, 배송비, 쿠폰, 포인트, 세금 정책은 따로 정의하지 않았으므로 현재 단계에서 상세 계산식은 보류한다.

*** 논리 테이블 설계

-> 각 테이블의 컬럼, PK, FK, nullable 여부, 부가 설명

    1. users 테이블 - 사용자 기준 엔티티

        사용자의 단위 행동 여정, 세션, 주문, 리뷰를 연결하는 기준

        column          key         nullable    discription

        **user_id       PK          N           사용자 식별자

        signup_date     -           Y           가입일

        user_type       -           N           회원/비회원 또는 신규/기존 사용자 구분

        age_group       -           Y           연령대. exact age 대신 범주형으로 관리

        gender          -           Y           성별. 분석 목적상 선택 속성

        -> user_id가 여러 테이블에서 참조됨.

        sessions.user_id, event_log.user_id, orders.user_id, reviews.user_id


    2. sessions 테이블 - 한번의 방문 단위

        퍼널 분석과 이탈 분석에 이용

        column              key     nullable    description

        session_id          PK      N           세션 식별자

        user_id             FK      N           세션을 발생시킨 사용자

        session_start_time  -       N           세션 시작 시각

        session_end_time    -       Y           세션 종료 시각

        traffic_source      -       Y           유입 출처

        medium              -       Y           유입 매체

        campaign            -       Y           캠페인명

        device_type         -       Y           기기 유형

        platform            -       Y           웹/앱 구분

        landing_page        -       Y           첫 진입 페이지

        -> FK : sessions.user_id -> users.user_id (1:N)

        한 사용자는 여러 세션을 가질 수 있다. 하나의 세션은 한 사용자에게 속한다.

    
    3. event_logs 테이블

        사용자의 행동 이벤트를 저장하는 중심 로그 테이블

        column              key     nullable    description

        event_id            PK      N           이벤트 식별자

        event_name          -       N           이벤트 이름

        event_timestamp     -       N           이벤트 발생 시각

        event_date          -       N           이벤트 발생 날짜

        user_id             FK      N           이벤트를 발생시킨 사용자

        session_id          FK      N           이벤트가 속한 세션

        product_id          FK      Y           상품 관련 이벤트에서 참조하는 상품

        category_id         FK      Y           상품 목록/카테고리 관련 이벤트에서 참조하는 카테고리

        order_id            FK      Y           구매 이벤트에서 참조하는 주문

        review_id           FK      Y           리뷰 작성 이벤트에서 참조하는 리뷰

        search_term         -       Y           검색 이벤트의 검색어

        list_id             -       Y           상품 목록 식별자 또는 목록 유형

        page_number         -       Y           검색/목록 페이지 번호

        quantity            -       Y           장바구니 추가 수량

        price               -       Y           이벤트 발생 시점의 상품 가격

        items_count         -       Y           결제 시작 시점의 상품 개수

        total_value         -       Y           결제/구매 관련 총 금액

        payment_method      -       Y           결제 수단

        rating              -       Y           리뷰 평점

        review_length       -       Y           리뷰 길이

        -> 모든 이벤트들이 한 테이블에 존재하기 떄문에 nullable이 많음.

        * FK 관계 정리
        
        evevent_logs.user_id    ->   users.user_id

        event_logs.session_id   ->   sessions.session_id

        event_logs.product_id   ->   products.product_id

        event_logs.category_id  ->   categories.category_id

        event_logs.order_id     ->   orders.order_id

        event_logs.review_id    ->   reviews.review_id

    ** 이러한 구조는 이벤트 종류가 많아지면 NULL 컬럼으로 인해 비효율적임.

    추후에 이벤트 파라미터 구조가 복잡해질 시에 event_params 또는 이벤트 별 상세 테이블 분리를 검토할 예정.


------------------------------------------------------------------------------------

    4. categories - 상품 분류 기준 테이블

        카테고리별 조회, 장바구니, 구매, 리뷰 분석시 사용

        column              key     nullable    description

        category_id         PK      N           카테고리 식별자

        category_name       -       N           카테고리명

        parent_category_id  FK      Y           상위 카테고리 식별자

        -> 계층 구조를 둔다면 자기 참조 관계가 생길수도 있음.

        categories.parent_category_id -> categories.category_id


    5. products - 상품 정보를 관리하는 핵심 기준 테이블

        view_item, add_to_cart, order_items, reviews 에서 참조됨.

        column              key     nullable    description

        product_id          PK      N           상품 식별자

        product_name        -       N           상품명

        category_id         FK      N           상품이 속한 카테고리

        brand               -       Y           브랜드명

        price               -       N           현재 상품 가격

        created_at          -       Y           상품 등록 시각

        is_active           -       N           현재 판매 활성 여부

        FK : products.category_id -> categories.category_id (1:N)

        한 카테고리는 여러 상품을 가질 수 있다. 하나의 상품은 하나의 카테고리에 속한다.

        * 가격은 시점에 따라 다르기 때문에, 현재 기준과 구매 당시의 상품 가격이 다르다.

        * is_active는 과거에 판매했지만, 현재 판매 종료일수 있음. TRUE/FALSE로 구분 가능

    
    6. orders - 구매 완료로 생성된 주문 단위를 관리하는 테이블

        purchase 이벤트가 발생했을 때 연결되는 결과물.

        column              key     nullable    description

        order_id            PK      N           주문 식별자

        user_id             FK      N           주문한 사용자

        session_id          FK      N           주문이 발생한 세션

        order_timestamp     -       N           주문 발생 시각

        total_value         -       N           주문 총액

        payment_method      -       Y           결제 수단

        order_status        -       N           주문 상태

        FK : orders.user_id -> users.user_id (1:N)

        orders.session_id -> sessions.session_id (1:N)

        한 사용자는 여러 주문을 할 수 있다. 
        
        한 세션에서 0개 이상의 주문이 발생할 수 있다.

    
    7.order_items - 주문 안에 포함된 개별 상품 행을 관리

        column              key     nullable    description

        order_item_id       PK      N           주문 상품 행 식별자

        order_id            FK      N           주문 식별자

        product_id          Fk      N           상품 식별자

        quantity            -       N           구매 수량

        item_price          -       N           구매 당시 상품 단가

        discount_amount     -       Y           상품 단위 할인 금액

        FK : order_items.order_id -> orders.order_id (1:N)

        order_items.product_id -> products.product_id (1:N)

        한 주문은 여러 주문 상품 행을 가진다.

        한 상품은 여러 주문 상품 행에 포함될 수 있다.

        * payment_method는 결제 수단.

            card, bank_transfer, kakao_pay, naver_pay

        * order_status는 주문 상태

            paid, cancelled, refunded, completed

            -> 구매 완료 주문만 분석할지, 취소 주문을 제외할지, 환불 주문을 별도로 볼지 고민해야함.


    8. reviews - 구매 이후 사용자가 작성한 리뷰 정보를 관리

        column              key     nullable    description

        review_id           PK      N           리뷰 식별자

        user_id             FK      N           리뷰 작성자

        product_id          FK      N           리뷰 대상 상품

        order_id            Fk      N           리뷰와 연결된 주문

        rating              -       N           리뷰 평점

        review_length       -       Y           리뷰 길이

        created_at          -       N           리뷰 작성 시각

        FK : reviews.user_id -> users.user_id (1:N)

        reviews.product_id -> products.product_id (1:N)

        reviews.order_id -> orders.order_id (1:N)

        한 사용자는 여러 리뷰를 작성할 수 있다.

        한 상품은 여러 리뷰를 받을 수 있다.

        한 주문에서 여러 상품 리뷰가 작성될 수 있다.

        * 추가적인 조건

        (reviews.order_id, reviews.product_id) -> (order_items.order_id, order_items.product_id)

        리뷰는 해당 주문에 포함된 상품에 대해서만 작성되어야 한다.

        event_logs.review_id -> reviews.review_id

        *** rating과 review_length는 event_logs와 reviews에 존재할 수 있지만,

        원칙적으로 리뷰의 상세 정보이기 때문에 reviews에 두는게 좋음.

        단, event_logs에 같이 두면 리뷰 이벤트만 빠르게 분석할 수 있는 장점이 있음.

    ## 문서 목적

        - 이 문서는 로그 기반 이커머스 유저 여정 분석 프로젝트의 논리적 테이블 명세를 정리한다.

        - 앞서 정의한 이벤트, 엔티티, 관계, 정합성 검증 규칙을 바탕으로 각 테이블의 역할, 칼럼, 키,

        nullable 여부, 데이터 타입 후보를 명시한다.

        - 본 문서는 실제 MySQL `CREATE Table` 문을 작성하기 전 단계의 설계 문서이며,

        이후 `sql/schema.sql` 작성의 기준으로 사용한다.

    
    ## 설계 기준

        - DBMS는 MySQL을 기준으로 한다.

        - 이벤트 로그는 1차 설계에서 단일 `event_logs` 테이블로 관리한다.

        - 이벤트별 파라미터는 `event_logs`의 nullable 칼럼으로 관리한다.

        - 사용자 단위 분석 편의를 위해 `event_logs`에는 `user_id`와 `session_id`를 모두 저장한다.

        - 상세 정보는 `users`, `sessions`, `products`, `orders`, `reviews` 등 별도 엔티티 테이블에서 관리한다.

        - 정합성 검증은 `docs/data_quality_rules.md`에서 별도로 관리한다.
