## DL-001. 프로젝트 도메인

### 배경

유저 여정 분석을 수행하기 위해 분석 대상 도메인을 먼저 정의해야 한다. 도메인에 따라 주요 이벤트, 퍼널 단계, 핵심 지표가 달라진다.

### 선택지 A

종합 이커머스 도메인

- 다양한 상품 탐색 흐름을 포함할 수 있다.
- 검색, 목록 조회, 상품 상세, 장바구니, 구매로 이어지는 표준적인 퍼널을 설계하기 쉽다.
- 로그 기반 사용자 행동 분석 예제로 확장성이 높다.

### 선택지 B

특정 카테고리 이커머스 도메인

- 패션, 식품, 전자제품 등 특정 카테고리에 맞춘 분석이 가능하다.
- 도메인 특화 지표를 더 구체적으로 정의할 수 있다.
- 초기 설계 단계에서는 범위가 좁아질 수 있다.

### 최종 선택

종합 이커머스 도메인을 선택한다.

### 선택 이유

초기 프로젝트에서는 특정 상품군보다 보편적인 이커머스 사용자 여정을 먼저 정의하는 것이 적합하다. 종합 이커머스는 검색, 탐색, 상품 조회, 장바구니, 구매 등 핵심 이벤트를 모두 포함할 수 있어 유저 여정 분석 구조를 설계하기 좋다.

### 선택하지 않은 이유

특정 카테고리 이커머스는 도메인 특화 분석에는 유리하지만, 초기 프로젝트에서 이벤트 체계와 데이터 모델을 먼저 잡기에는 범위가 지나치게 좁아질 수 있다.

### 보류된 내용

- 특정 상품 카테고리별 세부 분석 여부
- 카테고리별 전환율 비교 분석 여부

### 후속 작업

- 종합 이커머스 기준 주요 이벤트 목록 정의
- 상품, 카테고리, 주문 관련 기본 엔티티 정의

## DL-002. 분석 접근 방식

### 배경

프로젝트의 핵심 분석 방식을 결정해야 한다. 사용자 행동을 어떤 단위로 관찰하고 분석할지에 따라 데이터 설계와 분석 쿼리의 구조가 달라진다.

### 선택지 A

로그 기반 사용자 행동 분석

- 이벤트 단위로 사용자 행동을 추적할 수 있다.
- 퍼널, 전환, 이탈, 세션 흐름 분석에 적합하다.
- 실제 서비스 분석 방식과 유사한 구조를 만들 수 있다.

### 선택지 B

집계 지표 기반 분석

- 일별 매출, 주문 수, 사용자 수 등 요약 지표를 빠르게 분석할 수 있다.
- 데이터 구조가 단순하다.
- 개별 사용자 여정과 이벤트 순서를 분석하기 어렵다.

### 최종 선택

로그 기반 사용자 행동 분석을 선택한다.

### 선택 이유

이 프로젝트의 목적은 사용자 여정을 재구성하고 전환 병목을 찾는 것이다. 이를 위해서는 집계된 결과보다 사용자 행동 로그가 필요하다. 로그 기반 접근은 이벤트 순서, 세션 흐름, 단계별 전환 분석에 적합하다.

### 선택하지 않은 이유

집계 지표 기반 분석은 전체 성과를 빠르게 파악하는 데는 유용하지만, 사용자가 어떤 행동을 거쳐 구매하거나 이탈했는지 설명하기 어렵다.

### 보류된 내용

- 집계 지표 테이블을 별도로 만들지 여부
- 최종 리포트에서 매출 지표를 포함할지 여부

### 후속 작업

- 이벤트 로그 스키마 정의
- 세션 단위 분석 기준 정의
- 퍼널 분석 쿼리 작성 계획 수립

## DL-003. 이벤트 설계 방식

### 배경

로그 기반 분석을 수행하려면 이벤트 이름과 이벤트 파라미터를 일관된 기준으로 설계해야 한다.

### 선택지 A

GA4 스타일 이벤트 설계

- 이커머스 이벤트 분석에 널리 쓰이는 구조를 참고할 수 있다.
- `view_item`, `add_to_cart`, `begin_checkout`, `purchase` 같은 표준 이벤트를 활용할 수 있다.
- 퍼널 분석과 이벤트 파라미터 설계에 적합하다.

### 선택지 B

프로젝트 전용 커스텀 이벤트 설계

- 프로젝트 요구사항에 맞춰 자유롭게 이벤트명을 만들 수 있다.
- 도메인 특화 이벤트를 세밀하게 정의할 수 있다.
- 표준 이벤트 체계와의 비교 가능성은 낮아질 수 있다.

### 최종 선택

GA4 스타일 이벤트 설계를 채택한다.

### 선택 이유

GA4 스타일 이벤트 설계는 이커머스 사용자 행동 분석에 필요한 핵심 이벤트를 이미 잘 포괄한다. 초기 설계에서 표준적인 이벤트명을 사용하면 분석 쿼리, 문서화, 향후 확장이 쉬워진다.

### 선택하지 않은 이유

프로젝트 전용 커스텀 이벤트 설계는 자유도가 높지만, 초기 단계에서 이벤트명이 과도하게 분산될 수 있고 분석 기준이 불명확해질 위험이 있다.

### 보류된 내용

- GA4 권장 이벤트를 어디까지 그대로 따를지
- 커스텀 이벤트 추가 기준
- 이벤트 파라미터 세부 명명 규칙

### 후속 작업

- 초기 이벤트 허용 목록 작성
- 이벤트별 필수 파라미터 정의
- 이벤트 정합성 검증 규칙 작성

## DL-004. 프로젝트 문서 관리 방식

### 배경

프로젝트 계획, 현재 설계, 일정, 결정사항을 한 문서에서 모두 관리할지, 의사결정 기록을 별도 문서로 분리할지 결정해야 한다.

### 선택지 A

`project_plan.md`와 `decision_log.md`를 분리한다.

- 프로젝트 계획과 의사결정 기록의 역할이 명확히 분리된다.
- 현재 상태와 과거 결정 맥락을 따로 추적할 수 있다.
- 설계 변경이 발생했을 때 변경 이유를 남기기 쉽다.

### 선택지 B

하나의 프로젝트 문서에서 모두 관리한다.

- 문서 수가 적어 관리가 단순하다.
- 초기에는 빠르게 작성할 수 있다.
- 시간이 지나면 계획, 일정, 의사결정 이력이 섞일 수 있다.

### 최종 선택

`project_plan.md`와 `decision_log.md`를 분리한다.

### 선택 이유

분석 프로젝트는 진행 중에 데이터 정의, 이벤트 설계, 분석 범위가 바뀔 가능성이 있다. 계획 문서와 의사결정 기록을 분리하면 현재 상태와 결정 이력을 함께 관리하면서도 문서의 목적을 명확히 유지할 수 있다.

### 선택하지 않은 이유

하나의 문서에서 모두 관리하면 초기에는 편하지만, 의사결정이 누적될수록 현재 계획과 과거 판단 근거가 섞여 문서 가독성이 떨어질 수 있다.

### 보류된 내용

- 추가 문서 분리 여부
- 이벤트 정의서를 별도 문서로 만들지 여부
- 데이터 사전을 별도 문서로 만들지 여부

### 후속 작업

- 프로젝트 계획 문서 주기적 업데이트
- 주요 설계 변경 시 의사결정 기록 추가

## DL-005. 초기 이벤트 범위

### 배경

GA4 스타일 이벤트 설계를 채택한 뒤, 초기 분석에서 사용할 이벤트 범위를 확정해야 한다. 이벤트 범위는 데이터 모델, 정합성 검증, 퍼널 분석 쿼리의 기준이 된다.

### 선택지 A

핵심 이벤트 8개를 초기 범위로 설정한다.

- `session_start`
- `search`
- `view_item_list`
- `view_item`
- `add_to_cart`
- `begin_checkout`
- `purchase`
- `review_write`

### 선택지 B

이벤트 범위를 더 넓게 설정한다.

- 페이지 조회, 카테고리 클릭, 상품 추천 노출, 쿠폰 적용, 결제 실패, 환불 등 추가 이벤트까지 포함한다.
- 더 다양한 행동을 분석할 수 있다.
- 초기 데이터 설계와 정합성 검증 범위가 커진다.

### 최종 선택

초기 이벤트 범위는 8개 핵심 이벤트로 확정한다.

### 선택 이유

8개 이벤트는 종합 이커머스의 핵심 사용자 여정을 충분히 표현한다. `session_start`, `search`, `view_item_list`, `view_item`은 유입과 탐색 흐름을 설명하고, `add_to_cart`, `begin_checkout`, `purchase`는 구매 퍼널을 구성한다. `review_write`는 구매 퍼널의 필수 단계가 아니라 구매 이후 사용자 행동을 보기 위한 확장 이벤트로 사용한다.

### 선택하지 않은 이유

초기부터 이벤트 범위를 넓히면 이벤트 정의, 파라미터 설계, 정합성 검증 쿼리가 복잡해진다. 현재 단계에서는 핵심 여정을 안정적으로 분석할 수 있는 최소 범위를 먼저 확정하는 것이 적합하다.

### 보류된 내용

- 페이지 조회 이벤트 포함 여부
- 카테고리 클릭 이벤트 포함 여부
- 쿠폰 적용, 결제 실패, 환불 이벤트 포함 여부
- 상품 추천 노출 및 클릭 이벤트 포함 여부
- 리뷰 텍스트 감성분석은 현재 범위에 포함하지 않고 향후 확장 가능 항목으로 검토

### 후속 작업

- 8개 이벤트별 필수 파라미터 정의
- 이벤트 허용 목록 기반 정합성 검증 쿼리 작성
- 이벤트별 퍼널 단계 매핑 정의
- `review_write`는 핵심 구매 퍼널에는 포함하지 않고, 구매 이후 행동 분석용 이벤트로 분리한다.

## DL-006. 이벤트 로그 사용자 식별자 저장 방식

### 배경

이벤트 로그에서 사용자 단위 분석을 수행하려면 각 이벤트가 어떤 사용자와 세션에 속하는지 식별할 수 있어야 한다. `session_id`만 저장하더라도 `sessions` 테이블을 통해 `user_id`를 추론할 수 있지만, 분석 쿼리의 편의성과 정합성 검증 기준을 함께 고려해야 한다.

### 선택지 A

`event_logs`에 `user_id`와 `session_id`를 모두 저장한다.

- 사용자 단위 로그 분석에서 `sessions` 테이블 조인 없이 바로 집계할 수 있다.
- 세션 단위 분석과 사용자 단위 분석을 모두 쉽게 수행할 수 있다.
- `event_logs.user_id`와 `sessions.user_id` 간 정합성 검증이 필요하다.
- `user_id`가 중복 저장되므로 저장값 불일치 가능성을 관리해야 한다.

### 선택지 B

`event_logs`에는 `session_id`만 저장하고 `user_id`는 `sessions` 테이블을 통해 추론한다.

- 사용자 식별자의 중복 저장을 줄일 수 있다.
- 이벤트 로그와 세션 정보 간 사용자 불일치 위험이 줄어든다.
- 사용자 단위 분석마다 `sessions` 테이블 조인이 필요하다.
- 세션 테이블 누락 또는 조인 실패 시 사용자 단위 분석이 어려워질 수 있다.

### 최종 선택

`event_logs`에는 `user_id`와 `session_id`를 모두 저장한다.

### 선택 이유

이 프로젝트는 로그 기반 사용자 여정 분석이 핵심이므로 사용자 단위 이벤트 흐름, 전환율, 이탈 분석을 자주 수행한다. `session_id`만으로도 `sessions` 테이블을 통해 `user_id`를 추론할 수 있지만, 사용자 단위 로그 분석 편의성을 위해 `user_id`를 이벤트 로그에 함께 저장한다.

### 선택하지 않은 이유

`session_id`만 저장하는 방식은 중복 저장을 줄이는 장점이 있지만, 사용자 단위 분석마다 세션 테이블 조인이 필요하다. 초기 분석 단계에서는 쿼리 단순성과 분석 편의성을 더 중요하게 본다.

### 단점 및 정합성 검증 필요성

`user_id`를 중복 저장하므로 `event_logs.user_id`와 `sessions.user_id`가 불일치할 수 있다. 따라서 정합성 검증 항목에 `event_logs.user_id`와 `sessions.user_id`가 반드시 일치해야 한다는 기준을 포함한다.

### 후속 작업

- `event_logs.user_id`와 `sessions.user_id` 일치 여부 검증 쿼리 작성
- `session_id` 누락 또는 세션 테이블 미매칭 이벤트 검증 기준 정의
